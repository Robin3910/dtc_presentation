### Agent 的原理

其实后来回想一下，原理也比较简单——不是说大模型的原理，而是指 Agent 的原理。大家可以想象一下：

> Agent 就是提前封装了一些提示词，在打开代码编辑器的时候，他主动去获取了我们文件的目录结构，并提前读好了我们的文件内容（代码）。在我和他对话的过程中，他就直接按照我们的意思去修改我们的文件，最终达到修改代码的目的。

此外，他还能帮我执行很多黑框框命令，用来做调试和测试。对于程序员来说，就好比一个非常好用的帮手。

在那之后，我基本不自己动手写代码了，Cursor 帮我解决了 70% 左右的工作，我只负责点检功能和测试，然后反馈给他，让他再继续给我改，改到我满意为止。

### Agent 的本质

其实，Cursor 代码编辑器就是 Agent 的雏形。Agent 相比于市面上最普遍的问答 AI，最本质的区别就在于，他可以直接操控电脑，通过黑框框命令行来编写程序脚本、控制文件夹、联网搜索等等，基本上涵盖了我们电脑上 90% 的操作。

**那这意味着什么呢？** 意味着我们很多日常工作都可以交给 Agent 去做：

- 写代码
- 作图
- 写文案
- 发布文章
- 发布视频

等一系列能想到的工作。

## Part 2：Agent 工作原理拆解，如何快速上手 Agent

> 一旦你学会了 Agent，你发现你就离不开它了。没有了它你会直接摆烂，因为前后工作效率的对比差别太大了，完全接受不了！

### 学习门槛

有系统了解过 Claude 的同学，刚开始学习的时候应该都会有这种感觉：怎么这么多要记住的概念，还需要记忆 Claude 的各种命令、目录结构、如何编排。

其实这对普通人就有一定的门槛了，面对这个黑框框，很多人容易望而却步，这也是我今天给大家推荐 Cursor 的原因——纯图形界面操作，基本的日常工作事项流程的开发都能满足条件。

### 核心心法

其实 Claude 的这些东西对于普通人来说，没那么重要。**能了解它能给我们完成哪些工作，达成哪些目的，才是最关键的，也是当下 ROI 最高的。**

你完全可以让 AI 帮你构思架构，你自己来设计文件夹的层级，并出一份说明书，专门解释每个文件夹是做什么的。

最终，每个文件夹都可以成为你的一个流程员工，需要做什么的时候找到对应的文件夹（Agent）去做就可以了。

你可以组装 A+B+C Agent 来完成一个任务。

### 效果与愿景

最终达到的效果就是，拼乐高积木一样。今天有新的任务要完成，可能以前没系统地搭建过这个任务的流程，如果未来经常要用到，那就可以想一想自己有哪些 Agent 员工可以用，让他们组合起来帮我完成一系列任务，达成目的，我做检查。

如果某些功能还不能满足条件，完全就可以通过自己来造（准确地说是通过引导 AI 来帮我制作工作流程），然后最终去达到目的。

### 核心原则

> **核心点**：将 Agent 的功能拆得足够细，足够精美。也就是程序设计中的高内聚和低耦合，方便后续复用和组装。

#### 如何创建自己的 Skill

**工具：`skill-creator`**

当现有 Skill 满足不了需求时，可以自己造一个。

**SKILL.md 标准格式：**

```yaml
---
name: my-skill-name
description: 简洁描述（<200字符）
---

# My Skill

## When to Use
（何时调用这个 Skill）

## Capabilities
- 能力1
- 能力2

## Usage Guidelines
（使用指南）
```

**创建步骤：**

1. 安装 skill-creator：`npx skills add owner/repo@skill-creator -g -y`
2. 告诉它你的需求：`使用 skill-creator 帮我实现一个 xxx`
3. 它会引导你回答几个问题
4. 自动生成 SKILL.md 文件
5. 测试并迭代优化

**核心原则**：高内聚低耦合，给 AI 写指令而非给人写文档。

## Part 3：DTC 场景案例拆分 + Demo 演示

### 案例：AI 素材审核助手——从每天 3 小时到一键自动

---

#### 背景：素材审核为什么是痛点

在 DTC 联盟营销的日常运营中，素材审核是每天都会遇到的高频任务。达人、联盟伙伴会持续不断地提交图片、视频、文案等内容，每一份素材都需要人工对照品牌规范逐项检查：

- 图片是否使用了品牌禁止的元素（竞品 logo、特定颜色、夸张特效）
- 文案是否符合广告法要求（极限词、虚假承诺）
- 视频封面是否符合尺寸和清晰度要求
- 素材是否涉及版权问题

一个中等规模的 DTC 品牌，每天可能收到 20-30 份素材，每份审核平均耗时 5～10 分钟。这意味着运营每天要花 **1.5～4 小时** 在机械的审核工作上，而且人工审核还容易因为疲劳、情绪导致标准不一致。

---

#### 解决思路：两步走，从小到大的演进路径

不追求一步到位，直接搭一个"完美的全自动化系统"。从最小可用的 Demo 起步，跑通后再逐步演进：

**第一步（轻量 Demo）：** 本地文件夹 + 单个脚本

- 运营将收到的达人素材放入本地文件夹（图片 + 文案 + 达人邮箱）
- 写一个本地程序，自动读取素材 → AI 审核 → 自动发邮件通知达人结果
- 目标：跑通核心流程，证明 AI 审核可行

**第二步（完整系统）：** 部署到服务器，做成标准化产品

- 达人通过网页上传素材，不再用邮箱/微信传输
- AI 自动审核，结果实时推送通知
- 运营通过管理后台查看所有素材的审核状态
- 通过云存储（腾讯云 COS / 阿里云 OSS）管理所有上传素材
- 目标：从个人工具变成团队级产品

---

#### 技术架构

**Step 1 架构（轻量 Demo）**

```
本地文件夹（达人素材）
    ↓
Python 脚本（读取文件夹 + 整合内容）
    ↓
AI 分析（读取品牌规范 + 审核素材 + 生成报告）
    ↓
自动发送邮件（通过/需修改/违规 → 达人邮箱）
```

> 这一步只需要：Python 脚本 + 三方中转 API Key + 邮件发送服务（如 SendGrid / QQ 企业邮箱）。**没有任何外部依赖，当天就能跑起来。**

#### 品牌审核规则库示例

提前在项目中维护一份品牌规范清单，AI 读取后作为审核依据：

```markdown
## 品牌素材审核规范 v2.0

### 一、图片规范
- [x] 禁止出现竞品 logo（华为、OPPO、小米、三星等）
- [x] 禁止使用红色为主色调（品牌主色为蓝绿渐变）
- [x] 禁止添加外部水印
- [x] 图片分辨率不低于 1080×1080px
- [x] 禁止使用拉伸、变形、过度美颜处理

### 二、文案规范
- [x] 禁止极限词：「第一」「最好」「国家级」「全网最低」等
- [x] 禁止虚假承诺：「保证美白」「一定有效」「无效退款」（需附资质）
- [x] 禁止夸大功效（需与备案功效一致）
- [x] 禁止联系方式（微信号、手机号、网址链接）

### 三、视频规范
- [x] 时长不超过 60 秒（联盟平台限制）
- [x] 前3秒必须出现品牌元素
- [x] 背景音乐需为免费商用或已购买版权
```

#### 实际使用效果 & 关键心法总结

**Step 1（本地 Demo）：适合跑通流程、验证想法**

测试了不同的模型：GPT-4o、glm-4.6、qwen3-vl-plus、kimi-2.6，跑下来国内模型kimi-2.6的性价比是最高的。
在满足功能的条件上，价格可以做到比较低。

但同时也发现，最终审核结果的好坏，非常依赖于我们定下的规则提示词和大模型的选择，并且需要给足AI引导，告诉他：什么是违规的。

比如vivo手机上zeiss镜头的logo，如果没告诉他zeiss和vivo是建立品牌合作关系的，那AI就会认为在素材的宣传图上有zeiss的LOGO，是违规的，需要整改。

最终提示词足够稳定之后，每天就可以把审核的事情交给机器人去做，人可以抽离出来去做其他的工作，等机器人审核结束之后，再回来看看结果。
随着提示词不断迭代，最终的审核效果会越来越好。

目前所消耗的时间相比于人工审核，可能不会更快，但是带来的优势在于人可以解放出来去做其他的事情，毕竟机器可以7x24一直在运作。80%的初次筛查交给AI做，人只做剩下20%模糊不清的地方。

**Step 2（完整系统）：从个人工具到团队级产品**

说一下整体的开发体验，来到step2之后，整体的页面、代码功能都是让AI生成的。
AI在5分钟内便完成了服务器接口开发、前端页面开发、数据库开发，虽然只是初稿，但相比于以前我们自己选型、搭建框架、复制粘贴，需要花上1-2个小时，再到基础功能的开发、接口设计、数据库开发等等，还要花去小一周的时间，整体效率上真的是质的飞跃。

针对于一些技术问题报错，程序员在有经验的情况下，处理起来确实会快一些。能快速地去定位问题。但是，普通人在有AI的基础上，就可以通过不断地和AI反馈当前报错的信息，去修正BUG。
整个过程中，需要经历很多轮对话，才能把一个产品慢慢做好。

大家可以看到，公司以前找程序员+美工做一个这样类似的网站，需要花费的时间可能是1-2周，项目花销就要花费好几千好几万。
现在AI能将UI和代码的工作承包下来，搭配一名员工在非常短的时间内，可能1-2天，就能完工，并且上线。投入的时间精力资源是非常不成对比的。

所以，在当下这个时代，能熟练掌握AI，就能通过AI去撬动非常大的杠杆去做事。

> **规则越细，AI 越准。** 审核规范不是写给 AI 看的，是写给自己梳理业务标准的。规范写得越详细，AI 执行越到位。

> **从小开始，边跑边改。** 不需要一上来就搭建完整的系统。先用本地脚本跑通核心流程，验证 AI 审核确实有效，再逐步扩展成团队级产品。Step 1 是 Step 2 的最小可行性验证。

---

#### 扩展方向

Demo 跑通之后，可以进一步扩展：

- **Step 1 → Step 2**：本地脚本部署到服务器，增加网页上传页面和管理后台
- **被动触发 → 主动监控**：接入达人后台，当有素材提交时自动触发审核流程
- **单品牌 → 多品牌矩阵**：建立多套规范库，切换审核标准应对不同品牌合作方
- **自动分发**：审核通过的素材，一键同步发布到联盟平台

## Part 4：如何组建团队自用的工作流

### 为什么要组建团队自用的工作流

> **学遍最新最强的 AI 工具，不如搭建好一条团队能用的工作流。** 会用 AI 工具是个人能力，搭建自动化工作流才是团队能力，两者差距巨大。

#### 1. AI 正在把工程师重新分层

在 OpenAI，95% 的工程师每天使用 Codex，接近 100% 的代码先由 AI 生成再由人审查。**使用 Codex 更频繁的工程师，提交的 PR 数量比同事高出 70%，而且差距还在持续扩大。** 工程师的角色正在从"写代码的人"转变为"管理 Agent 舰队的人"——同时盯着 10～20 条并行的 AI 任务线程，主要工作是引导、验收、兜底。

这背后是一个残酷的事实：**AI 不会平均抬升所有人。能拆需求、能控上下文、能调度多 Agent、能把验证闭环做扎实的人，一个人就能顶过去一个小团队。** 差距不是慢慢拉开的，而是被工具放大、被流程放大，最后变成一种很难追回的"复利"。

#### 3. 团队工具库的复利效应

组建团队自用的工作流，核心价值在于**构建可复利的资产**：

- **A 开发了工作流 1，B 开发了工作流 2，C 开发了工作流 3**
- 三个人开发的内容全部沉淀在团队公用仓库里
- 哪天 A 想用工作流 3 组装一个工作流 4，**直接复用，不需要重新开发**
- 每个人的贡献都是**复利叠加**——写一次，可以一直被团队使用

这就像写团队工作的 SOP：写一次，长期复用。**一个人做工具，所有人受益；一个人踩坑，团队一起规避。**

#### 4. 行业正在分化：早动手 vs 继续观望

跨境电商等行业已经出现明显分化：

- **一类人**：追工具、追热点，学了一个又一个 AI 产品
- **另一类人**：把工作流当作基础设施来建，持续积累团队资产

两三年后，这两类人的差距会越来越大。**团队自用工作流的真正意义，不是追逐某个最新的 AI 工具，而是把 AI 能力真正沉淀为团队的生产力。**

### Git 的重要性

> **把团队的工具库变成"可复利的资产仓库"，Git 是这一切的基础设施。**

#### Git 不只是代码仓库，是团队协作的骨架

团队自用工作流之所以能真正落地，核心靠的不是哪个工具、哪个 AI 产品，而是一个**所有人都愿意往里贡献、所有人都知道怎么从这里取用**的公用仓库。Git 就是这套协作体系的骨架。

它的核心价值在于三个层次：

**① 版本控制 —— 记录每一步探索**
每次工作流的迭代都有记录：为什么这样设计、中间踩过什么坑、最终选择了哪个方案。新人不必重走弯路，直接站在前人的肩膀上继续。

**② 分支隔离 —— 让实验安全发生**
开一个新分支实验新工作流，不会影响主流程的稳定运行。成功了合并进来，失败了删掉重来，成本极低。

**③ 协作复用 —— 避免重复造轮子**
A 用 Git 管理了一套素材审核工作流，B 用它管理了一套报告生成工作流，C 则沉淀了一套数据清洗模板。三人互不干扰，却能互相取用——这才是"团队工具库"真正运转起来的样子。

#### 迁移路径：从小开始，逐步沉淀

很多团队想做但一直没做的原因是想得太复杂。**最小起步路径**：

1. **建一个私有仓库**，命名为 `team-ai-workflows`
2. **从今天最常用的 1 个工作流开始**，把它的 prompt、配置、说明文档全部提交进去
3. **每次解决一个新问题**，就在仓库里新增一个对应的工作流
4. **3 个月后**，仓库就变成了一座团队智库

> 迁移最大的障碍不是技术，而是习惯。让每个团队成员养成"做完工作流就提交"的习惯，比选什么 Git 平台更重要。GitHub、GitLab、Gitee 都行，重要的是先把内容沉淀进去。

每次更新新的内容到团队协作空间之后，都可以让 AI 去针对新的内容去完善一下使用说明。后续的新人进来之后，就完全按照使用说明进行操作，只需要进行些许的培训，他就能上手使用 AI，并且一个人可以顶好多人用。因为 AI 大大提效了。

### Token 成本

> **AI 使用的真正成本瓶颈，往往不在模型单价，而在缓存命中率和工作流组织方式。**

日常使用中 Token 消耗量巨大，有便宜好用的渠道直接决定了最终效果能否持续稳定运行。

#### 1. 渠道选择

**首选中转站**（或某宝三方套餐），而非官方渠道。原因很现实：

- 国内科学上网 IP 频繁变动，Claude、GPT 官方渠道极易识别为风险账号并封号
- 住宅固定 IP 成本太高，与"花最少钱办最多事"的 ROI 原则相悖
- 第三方中转站价格往往只有官方的 20%～50%，且对 IP 稳定性要求更低

> **选渠道时除了看价格，还要确认四点**：模型版本是否降级、能否查看 usage 详情、高峰期是否稳定、是否有缓存支持。

#### 补充：国内大模型方案——性价比极高的替代选择

国内模型近年来价格持续走低，在很多场景下已经是比 GPT/Claude 更经济的选择：

| 模型 | 输入价格 | 输出价格 | 缓存命中 | 适合场景 |
|------|---------|---------|---------|---------|
| **DeepSeek V4-Flash** | ¥1/百万 | ¥2/百万 | ¥0.02/百万 | 分类、提取、简单改写（价格杀手） |
| **阿里 Qwen3.6-Plus** | ¥2/百万 | ¥12/百万 | ¥0.2/百万 | 编程、通用生产任务 |
| **腾讯混元 Hy3 Preview** | ¥1.2/百万 | ¥4/百万 | ¥0.4/百万 | 主力均衡选择，企业可开票 |
| **月之暗面 Kimi K2.6** | ¥6.5/百万 | ¥27/百万 | ¥1.1/百万 | 长文本理解、深度分析 |
| **智谱 GLM-5.1** | ¥6/百万 | ¥24/百万 | ¥1.3/百万 | 多模态、企业级应用 |

#### 2. 模型分级路由：最大的降本杠杆（节省 40～60%）

不同任务用不同级别的模型，是降本效率最高的方式（以claude举例）：

| 层级      | 模型代表                | 单价特征   | 适合场景           |
| ------- | ------------------- | ------ | -------------- |
| **轻量级** | Haiku / GPT-4o-mini | 单次成本最低 | 分类、提取、改写、简单问答  |
| **主力级** | Sonnet / GPT-4o     | 性价比最优  | 编程、分析、通用生产任务   |
| **旗舰级** | Opus / GPT-5        | 单次成本最高 | 复杂规划、深度推理、代码审查 |

**实操建议**：使用前期不必想太多，直接拿 Sonnet 覆盖所有场景，性价比最高。中转站对 Sonnet 和 Opus 的费率往往一致——**同样价格，直接用更强的模型更划算。**

一个具体案例：某客服系统将 80% 简单查询从旗舰模型切到轻量模型后，月成本从 $12,000 降至 $2,800，准确率仅下降 1.2%。

#### 3. Prompt 缓存：最被低估的省钱手段（节省 50～90%）

重复的系统提示词和固定上下文，每次请求都重新计算是巨大的浪费。缓存命中后价格直接打骨折：


| 平台                   | 缓存命中率 | 折扣力度              |
| -------------------- | ----- | ----------------- |
| **Anthropic Claude** | 命中时   | **0.1x（相当于 9 折）** |
| **OpenAI**           | 自动缓存  | **0.5x（五折）**      |
| **Google Gemini**    | 命中时   | **0.25x（七五折）**    |


**实操关键**：将静态内容（系统提示词、示例、少变的文档）放在 prompt 的**最前面**，才能触发缓存命中。

#### 4. Prompt 精简：减少 70% 的无效消耗


| 优化前                 | 优化后              |
| ------------------- | ---------------- |
| 冗长段落式描述（180 tokens） | 结构化指令（50 tokens） |


**核心原则**：

- 禁止 AI 生成无用填充词（"好的"、"我明白了"、"这是个好问题"）
- 只输出修改的代码块，不重复发送未改内容
- 复杂任务才需要解释，简单任务直接给指令
- 一次性说清需求，避免"挤牙膏"式反复对话

#### 5. 理解真正的成本结构：上下文才是隐形杀手

很多人只盯着模型单价，却忽略了真正的成本黑洞——**上下文累积**：


| 成本驱动因素  | 占比     | 优化方向           |
| ------- | ------ | -------------- |
| 上下文历史累积 | 40～50% | 定期开启新会话，避免无限膨胀 |
| 工具输出存储  | 20～30% | 清理不必要的命令输出     |
| 系统提示重发  | 10～15% | 启用提示缓存，对齐 TTL  |
| 不当模型选择  | 5～10%  | 使用分级路由         |


> **多轮对话中，即使未达到上下文窗口，模型性能也会在 60～70% 容量时开始下降。** 定期清理对话历史，用摘要代替完整记录，是控制成本最有效的日常习惯。
